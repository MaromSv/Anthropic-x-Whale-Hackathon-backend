  How "default install" works                               
                                                                      
  Yes, the core pack lives at the same endpoint as the optional ones —
   just flagged differently. Cleaner than a separate endpoint.        
                                                                      
  The is_core field on each manifest tells the app what to do:        
  
  // Pseudo-Kotlin install flow                                       
  val manifests = api.getPacks()  // GET /packs             
                                                                      
  // On first launch: auto-download all core packs                    
  manifests.filter { it.isCore }.forEach { manifest ->                
      val pack = api.getPack(manifest.packId)  // GET /packs/{id}     
      db.savePack(pack)                                               
  }
                                                                      
  // Show the rest as opt-in cards in the UI                          
  val optionalPacks = manifests.filter { !it.isCore }
  // → user picks "King's Day Amsterdam" → download & save            
                                                                      
  How RAG should use the data
                                                                      
  Two layers, suggested fast→slow fallback:                           
  
  1. Tag-filtered fast lane: when user is at King's Day, vector-search
   only docs where tags contains "kings_day_priority". ~12 docs, super
   fast retrieval.                                                    
  2. Full set fallback: if top similarity < threshold, drop the filter
   and search the full installed pack(s).                             
  
  The priority: "high" field is a second knob — useful for ranking    
  results when mixing packs.                                
                                                                      
  POIs (map pins)                                                     
  
  Plot every point_of_interest on the map, color/icon by kind:        
  - ehbo → red cross 🩹 (8 of these on King's Day)          
  - toilet → blue 🚽 (128)                                            
  - park_ride → green 🅿️  (6)                                
                                                                      
  For "where's the nearest X?" queries, sort POIs by haversine        
  distance from user GPS — no API call needed, all data is local.     
                                                                      
  Versioning / refresh                                                
                                                            
  Each manifest has a version field. Store it locally per pack. On    
  launch, refetch /packs and compare versions — if backend version >
  local, redownload. (We're at 1.0 across the board for the           
  hackathon.)

  Errors to expect

  - 404 → pack id doesn't exist                                       
  - 500 → only if a pack JSON is missing on the server (shouldn't
  happen post-deploy)                                                 
  - Render cold start → first call after idle returns slow but normal
  200, subsequent are fast
